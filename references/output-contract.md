# 调研输出契约

最终报告可以是中文 Markdown。不要把所有材料平均铺开，先给结论，再给证据和缺口。

- <5-10 个最重要事实，每条带证据或标注“推断/假设”>

## 结论标记

使用这些标记保持诚实：
- `事实`：直接来自代码、测试、配置、文档或命令输出。
- `推断`：由多个证据合成，但没有直接声明。
- `假设`：为继续分析临时采用，必须可被验证或推翻。
- `未知`：缺少证据，不能判断。

## 探针格式

每个下一步探针都写成可执行动作：

```markdown
- 探针：<要查明的问题>
  动作：<读文件/追符号/运行测试/加指标/做实验>
  成功标准：<什么证据能把状态从 未讨论/已分析 推进到 已决策/已验证>
```

## Artifact 规则

启动调研时先扫描工作区，查找已有的 `research/` 子目录和包含勘探 artifact 的目录。若发现候选，列出候选路径并询问用户选择；若没有发现候选，先创建 `research/<project-slug>-investigation-YYYYMMDD/`，再询问用户是否将其作为本次工作区。用户确认前不写任何调研内容。

确认工作区后，所有中间笔记、路径图、缺口表、实验记录和最终报告都写入同一目录。若用户拒绝自动创建的候选目录，等待用户指定已有目录或其他工作区；不要改写项目根目录，也不要散落写入多个目录。

不要在项目根目录散落多个临时 Markdown 文件。

## 优先级规则

优先调查满足任一条件的问题：
- 会改变持久化、可见性、一致性或恢复语义。
- 位于读写快路径并影响 P99/P999。
- 会在过载、重试、取消、重启、降级或扩容时放大。
- 阻塞 benchmark、论文 claim、重构边界或立项决策。
